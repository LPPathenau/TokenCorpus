## 一、标注概述

### 1.1 标注目标

对语料中每个"token"的出现进行多层次语义标注，包括：
1. **语义类别**（必填）：token的具体含义
2. **隐喻类型**（必填）：是否为隐喻用法
3. **隐喻细节**（选填）：源域和目标域
4. **标注备注**（选填）：标注依据

### 1.2 标注原则

| 原则 | 说明 |
|------|------|
| **一致性** | 相同语境应有相同标注 |
| **可解释性** | 每个标注应有明确依据 |
| **独立性** | 标注员独立判断 |
| **保守性** | 不确定时标注为"未知"类别 |

## 二、语义类别标注体系

### 2.1 完整标签体系

```
Token语义类别（T + 序号）
├── T1: 安全认证类
│   ├── T1a: 访问凭证（access token）
│   ├── T1b: 会话标识（session token）
│   ├── T1c: CSRF防护（CSRF token）
│   ├── T1d: 刷新凭证（refresh token）
│   └── T1e: Bearer令牌（bearer token）
│
├── T2: NLP处理类
│   ├── T2a: 文本单元（word token, text token）
│   ├── T2b: 子词单元（subword token, BPE token）
│   ├── T2c: 分词器输出（tokenizer）
│   └── T2d: 特殊Token（[CLS], [SEP], <s>, </s>）
│
├── T3: 编译器/编程语言类
│   ├── T3a: 词法单元（lexical token）
│   ├── T3b: 语法标记（syntax token）
│   └── T3c: 保留字（reserved token, keyword token）
│
├── T4: 区块链/加密货币类
│   ├── T4a: 加密代币（crypto token）
│   ├── T4b: 功能代币（utility token）
│   ├── T4c: 安全代币（security token）
│   └── T4d: 代币兑换（token swap）
│
├── T5: AI/LLM大模型类
│   ├── T5a: 处理单元（input token, output token）
│   ├── T5b: 计费单位（per token, token cost）
│   ├── T5c: 上下文限制（context token, max tokens）
│   └── T5d: 提示词元（prompt token, system token）
│
├── T6: 网络协议类
│   └── T6a: 控制信号（token ring, token bus）
│
├── T7: 历史/其他
│   └── T7a: 历史用法（token coin, token economy）
│
└── T0: 未分类/模糊
    └── T0a: 泛指/不明确
```

### 2.2 标签定义详表

| 标签 | 名称 | 定义 | 典型语境 | 关键词 |
|------|------|------|---------|--------|
| **T1a** | 访问凭证 | 用于API访问的授权凭证 | "Obtain an access token from the authorization server" | access token, bearer token |
| **T1b** | 会话标识 | 标识用户会话的唯一标识符 | "The session token expires after 30 minutes" | session token, cookie token |
| **T1c** | CSRF防护 | 防止跨站请求伪造的令牌 | "Include the CSRF token in the form" | CSRF token, anti-forgery |
| **T1d** | 刷新凭证 | 用于刷新access token的凭证 | "Use the refresh token to get a new access token" | refresh token |
| **T2a** | 文本单元 | NLP处理中的基本文本片段 | "Each word is converted to a token" | word token, text token |
| **T2b** | 子词单元 | BPE等算法切分的子词 | "The tokenizer splits text into subword tokens" | subword, BPE, wordpiece |
| **T5a** | LLM处理单元 | 大模型处理的基本单元 | "The model can process up to 4096 tokens" | input token, output token |
| **T5b** | 计费单位 | API调用的计费单位 | "API calls are billed per token" | per token, token pricing |
| **T3a** | 词法单元 | 编译器的最小词法单位 | "The lexer produces lexical tokens" | lexical token, lexer token |
| **T4a** | 加密代币 | 区块链网络的数字资产 | "Transfer 100 tokens to the wallet" | crypto token, token transfer |
| **T6a** | 控制信号 | 网络协议中的访问控制信号 | "Token ring network uses token passing" | token ring, token bus |
| **T0a** | 泛指/模糊 | 语义不明确或泛指 | "Configure the token settings" | the token, any token |

## 三、隐喻类型标注体系

### 3.1 隐喻分类

| 类型 | 标签 | 定义 | 判断标准 |
|------|------|------|---------|
| **字面意义** | M0 | Token使用其基本或原初含义 | 无隐喻映射 |
| **结构隐喻** | M1 | 用一个概念结构理解另一个 | CONCRETE→ABSTRACT映射 |
| **方位隐喻** | M2 | 用空间方位理解抽象概念 | UP/DOWN, IN/OUT等 |
| **本体隐喻** | M3 | 将抽象概念视为实体 | 容器化、数量化 |
| **未知** | M4 | 无法判断 | 缺乏足够语境 |

### 3.2 判断流程

```
┌─────────────────────────────────────────────────────────────┐
│              Token隐喻类型判断流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: Token的原始含义是什么？                            │
│          └── 物理凭证/通行证（现实世界的实物）             │
│                           ↓                                 │
│  Step 2: 当前语境中Token是否指现实世界的实物？             │
│          ├── 是 → M0 (字面意义)                            │
│          └── 否 → ↓                                        │
│                                                             │
│  Step 3: 是否用"凭证/通行证"的概念理解当前语境？         │
│          ├── 是 → M1 (结构隐喻)                            │
│          │   └── 追问：源域→目标域是什么？                │
│          └── 否 → ↓                                        │
│                                                             │
│  Step 4: 是否用空间/容器概念理解Token？                   │
│          ├── 是 → M2 (方位隐喻)                            │
│          └── 否 → ↓                                        │
│                                                             │
│  Step 5: 是否将Token概念实体化？                           │
│          ├── 是 → M3 (本体隐喻)                            │
│          └── 否 → M4 (未知)                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、标注决策指南

### 4.1 安全认证类 vs NLP处理类

**关键区分特征**：

| 特征 | T1 (安全认证) | T2 (NLP处理) |
|------|---------------|--------------|
| 周围词 | authenticate, authorize, credential | tokenizer, tokenize, vocabulary |
| 搭配 | obtain, request, issue, revoke | split, convert, encode |
| 上下文 | security, API, authentication | NLP, model, text processing |
| 可数性 | 通常单数，但可复数 | 常与数量词连用 |

**示例判断**：

```
句子1: "The API requires an access token."
判断: T1a (安全认证-访问凭证)
理由: 搭配"API requires"表明这是认证场景

句子2: "The text is tokenized into word tokens."
判断: T2a (NLP-文本单元)
理由: 搭配"tokenized into"表明这是NLP处理场景
```

### 4.2 NLP处理类 vs AI/LLM类

**关键区分特征**：

| 特征 | T2 (NLP处理) | T5 (AI/LLM) |
|------|-------------|-------------|
| 时代背景 | 传统NLP (2015年前) | 大模型时代 (2018年后) |
| 典型词 | word, sentence, corpus | GPT, LLM, transformer, context |
| 数量语境 | 通常讨论tokenization方法 | 常与计费、上下文限制相关 |
| 技术语境 | 预处理、分词算法 | API调用、模型推理 |

**示例判断**：

```
句子1: "We use BPE tokenization to split text into subword tokens."
判断: T2b (NLP-子词单元)
理由: 讨论tokenization方法，传统NLP语境

句子2: "GPT-4 can process up to 128K tokens in a single request."
判断: T5a (LLM-处理单元)
理由: 讨论大模型上下文窗口，LLM语境
```

### 4.3 区块链代币 vs 安全凭证

**关键区分特征**：

| 特征 | T4 (加密代币) | T1 (安全凭证) |
|------|--------------|---------------|
| 周围词 | blockchain, wallet, transfer, swap | authenticate, authorize, security |
| 搭配 | mint, burn, stake, governance | obtain, refresh, revoke |
| 技术栈 | Ethereum, smart contract, DeFi | OAuth, JWT, API |
| 可替换性 | 通常可交易，有经济价值 | 通常绑定身份，不可交易 |

## 五、标注流程

### 5.1 标准标注流程

```
┌─────────────────────────────────────────────────────────────┐
│                    标注工作流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 阅读句子                                               │
│     └── 完整理解句子含义                                    │
│                           ↓                                 │
│  2. 定位Token出现                                          │
│     └── 找到句子中的"token"及其相关词                       │
│                           ↓                                 │
│  3. 分析语境                                               │
│     ├── 周围关键词                                          │
│     ├── 来源领域                                            │
│     └── 语义特征                                          │
│                           ↓                                 │
│  4. 判断语义类别                                           │
│     └── 对照标签定义，选择最合适类别                        │
│                           ↓                                 │
│  5. 判断隐喻类型                                           │
│     └── 使用决策流程图                                      │
│                           ↓                                 │
│  6. 填写备注（如有）                                       │
│     └── 记录判断依据                                        │
│                           ↓                                 │
│  7. 复核                                                   │
│     └── 检查标注一致性                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 歧义处理

**规则**：
1. 优先参考来源领域标签（source_type）
2. 结合关键词和搭配综合判断
3. 无法判断时标注为T0（未分类）
4. 记录歧义案例用于讨论

**示例歧义处理**：

```
句子: "The token contains user information."
歧义点: 可能是JWT token（T1），也可能是数据序列化（T2）

处理方案：
- 查看来源：如来自OAuth文档 → T1a
- 查看周围词：如包含"sign" → T1a
- 如仍无法判断 → 标注T0 + 备注"歧义：可能为JWT或数据格式"
```

## 七、质量控制

### 7.1 标注培训

| 阶段 | 内容 | 时长 |
|------|------|------|
| 规范讲解 | 标签定义、判断规则 | 1小时 |
| 示例练习 | 10个黄金标准例 | 30分钟 |
| 试标注 | 50条语料独立标注 | 1小时 |
| 结果讨论 | 一致性分析、问题解答 | 30分钟 |
| 正式标注 | 开始正式标注 | - |

### 7.2 质量指标

| 指标 | 目标值 | 计算方法 |
|------|--------|---------|
| 标注一致性 | ICC ≥ 0.85 | Krippendorff's Alpha |
| 标注准确率 | ≥ 90% | 与黄金标准对比 |
| 标注速度 | 30-50条/小时 | 时间统计 |

### 7.3 抽检机制

- **每日抽检**：随机抽检当天标注的5%
- **每周校准**：召开标注一致性会议
- **月度评估**：计算整体ICC，更新规范
